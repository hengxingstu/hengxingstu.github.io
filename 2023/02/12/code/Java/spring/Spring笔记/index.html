Spring

一、引言

1.1 原生web开发中存在哪些问题？

  • 传统Web开发存在硬编码所造成的过度程序耦合（例如：Service中作为属性Dao对象）。

  • 部分Java EE API较为复杂，使用效率低（例如：JDBC开发步骤）。

  • 侵入性强，移植性差（例如：DAO实现的更换，从Connection到SqlSession）。

二、Spring框架

2.1 概念

  • Spring是一个项目管理框架，同时也是一套Java EE解决方案。

  • Spring是众多优秀设计模式的组合（工厂、单例、代理、适配器、包装器、观察者、模板、策略）。

  • Spring并未替代现有框架产品，而是将众多框架进行有机整合，简化企业级开发，俗称”胶水框架”。

2.2 访问与下载

官方网站：https://spring.io/

下载地址：http://repo.spring.io/release/org/springframework/spring/

2.3 Spring架构组成

Spring架构由诸多模块组成，可分类为

  • 核心技术：**依赖注入，事件，资源，i18n，验证，数据绑定，类型转换，SpEL，AOP**。
  • 测试：模拟对象，TestContext框架，Spring MVC测试，WebTestClient。
  • 数据访问：事务，DAO支持，JDBC，ORM，封送XML。
  • Spring MVC和 Spring WebFlux Web框架。
  • 集成：远程处理，JMS，JCA，JMX，电子邮件，任务，调度，缓存。
  • 语言：Kotlin，Groovy，动态语言。

Bean注册的细节

概念

DI（Dependency Injection）依赖注入【重点】

在Spring创建对象的同时，为其属性赋值，称之为依赖注入。

  • 构造方法注入
  • set 方法注入（推荐）
  • p名称空间注入（本质上还是 set 方法注入）

Warning 注意属性名

内省。

对于框架而言，并不是看对象定义的属性叫什么名字，而是根据对象的get/set方法来推断属性名称。无论是Mybatis、Spring、SpringMVC、所有框架，只要用到反射都是这样的。

所以，定义get/set方法的时候，不要写错，另一方面，变量的命名要复合规范。

Java代码配置Spring

/**
 * java配置类，它的作用类似于applicationContext.xml
 * @Configuration 表示这是一个配置类
 */
@Configuration
public class JavaConfig {

    /**
     * @Bean 就表示将当前方法的返回值注册到Spring容器中
     * 默认情况下，方法名就是bean的名字
     * 如果要自定义名字，就在注解中配置@Bean("你的名字")
     */
    @Bean("u")
    User user(){
        User user = new User();
        user.setName("琪亚娜");
        user.setId(001);
        return user;
    }
}

然后你的启动类也需要一点点改变，创建AnnotationConfigApplicationContext对象来接收参数

public class Demo01 {

    }
}

参数自动注入

现在我们在配置类中注册Bean，注意这里的book有一项属性为author类

@Configuration
public class JavaConfig {

    /**
     * @Bean 就表示将当前方法的返回值注册到Spring容器中
     */
    @Bean
    Author author(){
        Author author = new Author();
        author.setName("太宰治");
        author.setAge(23);
        return author;
    }

    @Bean("book1")
    Book book(){
        Book book = new Book();
        book.setId(001);
        book.setName("人间失格");
        book.setAuthor(author());//在这里直接调用bean的注册方法
        return book;
    }
}

这种情况下，book1的作者是之前创造的author，还是重新创造了一个？我们一试便知。

public static void main(String[] args) {
    AnnotationConfigApplicationContext acc = new AnnotationConfigApplicationContext(JavaConfig.class);
    Book book1 = acc.getBean("book1", Book.class);
    System.out.println(book1);//Book{id=1, name='人间失格', author=Author{name='太宰治', age=23}}

    Author author = acc.getBean("author", Author.class);
    System.out.println(author);//Author{name='太宰治', age=23}
    System.out.println("book1.getAuthor() == author : " + (book1.getAuthor() == author));
    //book1.getAuthor() == author : true
}

说明author对象是一致的

  • @Configuration：这个注解表示当前类是一个配置类，那么当前类中，所有添加了@Bean注解的方法都会被注册到Spring容器中，如果有其他方法调用到一个添加了@Bean注解的方法，那么不会立马执行对应的方法，而是先去Spring容器中查看是否有对应的对象。如果有，则直接从容器中获取，没有的话才会执行对应的方法。

  • @Component虽然也可以加载配置类上，但是，如果有其他方法掉哟共到一个添加了@Bean注解的方法，那么不会去检查是否有相应对象了，而是直接执行对应方法。

    上面的情况换成@Component最终的结果就是false

    如果一定要使用@Component，可以通过依赖注入来代替方法调用，修改注册book1的代码

    @Bean("book1")
Book book(Author author){//将author作为形参传入
    Book book = new Book();
    book.setId(001);
    book.setName("人间失格");
    book.setAuthor(author);
    return book;
}

所以我们在配置类中一般不用@Component

条件注解

条件注解的作用是告诉Spring在某些条件下再执行某些操作。

我们来实现一个功能，当系统检测到当前系统是Windows，系统就输出dir，是Linux时就输出ls。

实现方式

通过Spring提供的Condition接口中的match方法，可以获得系统的参数，再利用@Condition当返回确定是某个系统的信息后注册对应的Bean，并返回信息。

先写一个接口，确定输出系统信息

public interface ShowCmd {
    String cmd();
}

实现对应两个系统的类

//win
public class WindowsShowCmd implements ShowCmd {
    @Override
    public String cmd() {
        return "dir";
    }
}
//linux
public class LinuxShowCmd implements ShowCmd {
    @Override
    public String cmd() {
        return "ls";
    }
}

通过Spring提供的接口获取系统信息

import org.springframework.context.annotation.Condition;
//注意这里的接口是Spring中的Condition
//win
public class WindowsCondition implements Condition {
    public boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
        //获取当前操作系统的名字
        String property = context.getEnvironment().getProperty("os.name");
        return property.toLowerCase().contains("win");
    }
}
//linux
public class LinuxCondition implements Condition {
    public boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
        String property = context.getEnvironment().getProperty("os.name");
        return property.toLowerCase().contains("linux");
    }
}

配置类中注册对应bean，使用@Conditional(LinuxCondition.class)，当这个类的matches()方法返回true，这个bean就会被注册，反之不会。

@Configuration
public class JavaConfig {
    /**
     * 向 Spring 容器中注册一个 Bean
     *
     * @Conditional(WindowsCondition.class) 就是一个条件注解，如果里边的类的 matches 方法返回 true，就表示条件满足，这个 bean 就会注册到 Spring 容器中，如果返回 false，那么这个 Bean 就不会注册到 Spring 容器中。
     */
    @Conditional(LinuxCondition.class)
    @Bean
    ShowCmd linuxCmd(){
        return new  LinuxShowCmd();
    }

    @Conditional(WindowsCondition.class)
    @Bean
    ShowCmd WinCmd(){
        return new WindowsShowCmd();
    }
}

现在我们通过启动类检测一下：

public static void main(String[] args) {
    AnnotationConfigApplicationContext acc = new AnnotationConfigApplicationContext(JavaConfig.class);

    ShowCmd cmd = acc.getBean(ShowCmd.class);
    System.out.println(cmd.cmd());//dir	我当前是Windows系统
}

成功了

多环境切换

通过Java代码实现

比如生产环境和开发环境间切换

我们设置一个数据库的信息类

public class DataSource {
    private String name;
    private String password;
    private String url;
    //getter、setter等方法
}

在配置类中，我们分别设置生产和开发两个Bean。通过@Profile("dev")注解可以指定当前Bean在哪个环境下生效

@Configuration
public class JavaConfig {

    @Profile("dev")
    @Bean
    DataSource devDataSource(){
        DataSource dataSource = new DataSource();
        dataSource.setName("hengxing");
        dataSource.setPassword("123123");
        dataSource.setUrl("localhost:3306:///test1");
        return dataSource;
    }

    @Profile("prod")
    @Bean
    DataSource prodDataSource(){
        DataSource dataSource = new DataSource();
        dataSource.setName("tt");
        dataSource.setPassword("123");
        dataSource.setUrl("12.12.12.12:3306:///prod");
        return dataSource;
    }
}

接下来就可以去获取，注意先不要写配置类，要先设置环境信息。设置好后再填入配置类，刷新。

public static void main(String[] args) {
    //注意先不要写配置类，要先设置环境信息
    AnnotationConfigApplicationContext acc = new AnnotationConfigApplicationContext();
    //设置当前的环境信息
    acc.getEnvironment().addActiveProfile("dev");//设置当前环境为dev，这里的dev就是profile中设置的
    acc.register(JavaConfig.class);
    acc.refresh();//刷新

    DataSource dataSource = acc.getBean(DataSource.class);
    System.out.println("dataSource = " + dataSource);
    //dataSource = DataSource{name='hengxing', password='123123', url='localhost:3306:///test1'}
}

Spring是怎么实现的？

我们来看Profile注解的代码，里面有一个属性，注意它的意思可不是一个方法，而是一个键值对，键为value，值为一个String类型的数组。

@Conditional(ProfileCondition.class)
public @interface Profile {

   /**
    * The set of profiles for which the annotated component should be registered.
    */
   String[] value();

}
class ProfileCondition implements Condition {

   @Override
   public boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) {
      MultiValueMap<String, Object> attrs = metadata.getAllAnnotationAttributes(Profile.class.getName());
      if (attrs != null) {
         for (Object value : attrs.get("value")) {//获取键为value的属性
            if (context.getEnvironment().acceptsProfiles(Profiles.of((String[]) value))) {
                //判断当前环境中有没有value中的值。
               return true;
            }
         }
         return false;
      }
      return true;
   }

}

Java代码配置

@PropertySource("classpath:db.properties") 当容器启动的时候，指定配置文件的位置，那么配置文件中的内容，就会被自动的注入到 Spring 容器中

与@Value("${db.username}")结合使用。

@Value注解是Spring框架中的一个常用功能,其作用是通过注解将常量、配置文件中的值、其他bean的属性值注入到变量中，作为变量的初始值。

详细资料：关于Spring中@Value注解使用

@Configuration
@PropertySource("classpath:db.properties")//注入到 Spring 容器中
public class DSConfig {

    //跟 Spring 容器要一个字符串回来
    @Value("${db.username}")
    String username;
    @Value("${db.password}")
    String password;
    @Value("${db.url}")
    String url;

    @Bean
    DataSource dataSource(){
        DataSource dataSource = new DataSource();
        dataSource.setName(username);
        dataSource.setPassword(password);
        dataSource.setUrl(url);
        return dataSource;
    }
}

接下来通过Spring容器获取bean就行了

public class StartWithJavaConfig {
    public static void main(String[] args) {
        AnnotationConfigApplicationContext context = new AnnotationConfigApplicationContext(DSConfig.class);
        DataSource source = context.getBean(DataSource.class);
        System.out.println("source = " + source);
        //source = DataSource{name='root', password='1234', url='jdbc:mysql:///db?serverTimezone=Asia/Shanghai'}
    }
}

包扫描

之前注册Bean都是一个一个来的，效率太低了，有没有办法让它自动注册？

以一个典型的Servlet工程为例

上图是它的工程结构，如果我们使用之前注册Bean的方法就太慢了，现在来演示如何让其自动注册

Java代码配置包扫描

整体上来说，有三个方面：

  1. 将Bean注册到Spring容器中：@Repository、@Service、@Controller、@Component
  2. 从Spring容器中要一个Bean回来。官方推荐的方式，是通过构造器来注入。
  3. 配置包扫描

需要我们给需要注册的bean加上注解

/**
 * 将 UserDao 注册到 Spring 容器中，将来谁需要使用 UserDao，就直接跟 Spring 容器去查找即可
 *
 *
 * @Repository：一般是加在 Dao 层
 * @Service：一般是加在 Service 层
 * @Controller：一般是加在控制层，也就是 servlet
 * @Component：身份不明的 Bean 注册到 Spring 容器中时，使用这个注解
 *
 * 技术上来说，这四个注解并没有差异，即实际上除了 @Controller 之外，其他几个注解是可以混用的。
 *
 * 实际开发中，不要混用
 *
 */
@Repository
public class UserDao {
    public User getUserByUsername(String username){
        User user = new User();
        user.setName(username);
        user.setAddress("陕西");
        return user;
    }
}

这里又设计一个问题，如果有多个构造器怎么办？@Autowired注解可解决

/**
 * UserService 想要使用 UserDao
 * 1. UserService 自己需要在 Spring 容器
 * 2. 跟 Spring 容器去要一个 UserDao
 */
@Service
public class UserService {
    UserDao userDao;

    /**
     * 官方推荐使用构造器注入。因为 UserService 要注册到 Spring 容器，要生成一个对象，必然就要调用它的构造方法，现在它只有这一个构造法方法，那么 Spring 容器只能通过这个构造方法来获取一个 UserService 的实例，那么 Spring 容器会自动去查找容器中是否有一个 UserDao 的实例，如果有，就作为这个方法的参数传入进来
     * 如果有多个构造方法，那么可以通过 @Autowired 注解来告诉 Spring 容器，应该调用哪一个构造方法去初始化当前对象
     */
    public UserService(UserDao userDao){
        this.userDao = userDao;
    }

    public User getUserByName(String name){
        return userDao.getUserByUsername(name);
    }
}
@Controller
public class UserServlet {
    UserService userService;

    public UserServlet(UserService userService){
        this.userService = userService;
    }

    public User getUserByName(String name){
        return userService.getUserByName(name);
    }
}

配置包扫描

@Configuration
@ComponentScan(basePackages = "com.hengxing.demo")
public class JavaConfig {
}

AnnotationConfigApplicationContext context = new AnnotationConfigApplicationContext(JavaConfig.class);

日常使用较多的写法

与前面的步骤相比，主要是第二步不一样，其他都差不多

@Controller
public class UserServlet {
    /**
     * @Autowired 表示根据类型去 Spring 容器中查找到相应的 Bean，并赋值给 userService 变量
     */
    @Autowired
    UserService userService;

    public User getUserByName(String name){
        return userService.getUserByName(name);
    }
}

你可以注意到我们不需要写构造方法，或者也可以认为使用了默认的空参构造。在注册了Bean后，直接从容器中查找。那为什么我们使用这种方式？

来对比两者的区别：

  • 构造方法注入：

    • 官方考虑的是，这个类将来使用的时候，用户不一定是和Spring要，而有可能是自己new出来。

      假如此时不止一个构造器，一个空参，一个有参。用户直接通过空参new一个UserServlet，那此时其中的userService变量就为null，并且用户没有办法给userService赋值，导致userService调用时空指针。

      假如此时只有一个有参构造器，用户自己new的时候也需要传一个userService进去，就不会导致空指针了。

    • 但是实际开发环境中，一个类的属性会特别多，导致其构造方法太长。所以一般不这样写。

  • 注解注入：

    • 简单便捷
    • 如果用户使用不规范，自己new对象，就会有很多属性没有赋值。

工厂Bean

这主要是解决一些第三方的Bean，一些无法通过构造方法正常初始化、或者无法通过set方法正常为属性赋值的Bean，可以通过工厂Bean的方式将之注册到Spring容器中。

我们在项目中使用OkHttp做网络客户端，访问百度网页。这个包的依赖：

<dependency>
    <groupId>com.squareup.okhttp3</groupId>
    <artifactId>okhttp</artifactId>
    <version>3.14.2</version>
</dependency>
public class Demo01 {
    public static void main(String[] args) {
        //这个的作用，就是用来发送一个一个的请求
        OkHttpClient httpClient = new OkHttpClient.Builder()
                //设置服务端的读取超时时间
                .readTimeout(5000, TimeUnit.SECONDS)
                //连接超时
                .connectTimeout(5000,TimeUnit.SECONDS)
                .build();
        //构建一个具体的请求
        Request request = new Request.Builder()
                //设置请求方法是 get
                .get()
                //设置请求的 URL 地址
                .url("http://www.baidu.com").build();
        Call call = httpClient.newCall(request);
        call.enqueue(new Callback() {
            //请求失败的回调
            public void onFailure(Call call, IOException e) {

            }

            //请求成功的回调
            public void onResponse(Call call, Response response) throws IOException {
                System.out.println(response.body().string());
                //我们把网页打印出来就行
            }
        });
    }
}

以上代码经过验证，可以访问。

它在实例化时必须设置好参数，并且不可以通过构造方法修改参数，只能在实例化时调用某些方法来设置。然而Spring容器却都是通过构造方法来注册Bean的，难道就用不了了吗？

并不是，我们可以通过工厂创造这样的对象。工厂又分两种，静态工厂和实例工厂。

静态工厂

静态工厂中，工厂方法是一个静态方法。

public class OkHttpFactory {
    public static OkHttpClient okHttpClient;
    public static OkHttpClient getInstance(){
        if (okHttpClient == null){
            okHttpClient = new OkHttpClient.Builder()
                    //设置服务端的读取超时时间
                    .readTimeout(5000, TimeUnit.SECONDS)
                    //连接超时
                    .connectTimeout(5000, TimeUnit.SECONDS)
                    .build();
        }
        return okHttpClient;
    }
}

在applicationContext.xml中注册这个Bean，标识工厂方法。直接

<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="http://www.springframework.org/schema/beans http://www.springframework.org/schema/beans/spring-beans.xsd">

    <!--
    标识工厂方法
    这个 Bean，将来会自动调用 getInstance 方法，并将该方法的返回值注入到 Spring 容器中
    -->
    <bean class="com.hengxing.demo.OkHttpFactory" factory-method="getInstance"/>
</beans>
注意

这里将来注册到Spring容器中的Bean，不是class对应的类的对象，而是getInstance()方法返回的对象。

实例工厂

实例工厂就是一个实例化后的工厂，此时就不能使用静态方法的调用方式来注册了。必须要先有工厂对象，然后才能调用对应方法

这是工厂类的代码

public class OkHttpFactory01 {
    public OkHttpClient okHttpClient;
    public OkHttpClient getInstance(){
        if (okHttpClient == null){
            okHttpClient = new OkHttpClient.Builder()
                    //设置服务端的读取超时时间
                    .readTimeout(5000, TimeUnit.SECONDS)
                    //连接超时
                    .connectTimeout(5000, TimeUnit.SECONDS)
                    .build();
        }
        return okHttpClient;
    }
}

Bean的注册：

<?xml version="1.0" encoding="UTF-8"?>
<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="http://www.springframework.org/schema/beans http://www.springframework.org/schema/beans/spring-beans.xsd">

    <!--对于实例工厂方法，必须得先有一个工厂的实例，将来要通过这个实例才能调用工厂方法-->
    <bean class="com.hengxing.demo.OkHttpFactory01" id="factory01"/>
    <!--此时配置工厂的时候，既需要指定工厂的 Bean，也需要指定工厂方法-->
    <bean class="okhttp3.OkHttpClient" factory-bean="factory01" factory-method="getInstance" id="client2"/>
</beans>

Tips

这种方式获取的是工厂注册好的Bean那如果我想要工厂Bean呢？

在getBean()时在id前添加&符号即可，例如:

public class Demo04 {
    public static void main(String[] args) {
        ClassPathXmlApplicationContext context = new ClassPathXmlApplicationContext("applicationContext.xml");
        Object bean = context.getBean("&client3");
        System.out.println("bean.getClass() = " + bean.getClass());
        //bean.getClass() = class com.hengxing.demo.OkHttpFactoryBean
        //得到了工厂bean
    }
}

Bean的作用域

默认情况下，注册到Spring容器中的Bean是单例的：

<bean id="user" class="com.hengxing.scope.Model.User"/>

如果不希望这个Bean是单例的，可以通过如下方式修改：

<bean id="user" class="com.hengxing.scope.Model.User" scope="prototype"/>

scope有五种取值

  • singleton 表示这个Bean是单例的，每次取到的都是一个对象
  • prototype 非单例，每次取到都是新的
  • request 在同一个请求中，如果对此获取同一个Bean，取到的是同一个（Web下生效）
  • session 在同一个session中，如果对此获取同一个Bean，取到的是同一个（Web下生效）
  • application 在同一个Web应用中，如果对此获取同一个Bean，取到的是同一个（Web下生效）

也可以通过java代码配置scope，通过@Scope注解：

@Configuration
public class JavaConfig {

    @Bean
    @Scope("prototype")//字符串书写
    User user(){
        User user = new User();
        user.setName("hengxing");
        user.setAge(13);
        return user;
    }
}

Bean的初始化和销毁

如果在Spring容器完成注册之后，还想要做一些配置，那么可以在init方法中完成，当Spring容器销毁的时候，可以在destory方法中完成一些资源回收操作。

定义好初始化和销毁方法

public class User {
    private String name;
    private Integer age;

	public User() {
        System.out.println("构造方法。。。");
    }
    
    public void init(){
        System.out.println("初始化方法。。。");
    }

    public void destory(){
        System.out.println("销毁方法。。。");
    }

使用xml将其注册到Spring中去

<bean id="user" class="com.hengxing.scope.Model.User" init-method="init" destroy-method="destory"/>

现在用xml注册文件方式执行

public static void main(String[] args) {
    ClassPathXmlApplicationContext context = new ClassPathXmlApplicationContext("applicationContext.xml");
    User bean = context.getBean(User.class);
    System.out.println("bean = " + bean);
    context.close();//用来关闭spring容器
}

IoC小结

控制反转，指的是对象的控制权反转。

本来，UserService需要一个UserDao，那么直接在UserService中new一个UserDao，此时UserDao的对象控制权就在UserService中。

有了Spring之后，系统启动时，UserDao会将自己的各种信息告诉Spring容器，Spring利用这些信息，就可以结合反射创建一个UserDao对象，此时UserDao对象处于容器中，这个对象的控制权在容器中，==控制权==反转了。

利用IoC可以实现对象之间的解耦。
