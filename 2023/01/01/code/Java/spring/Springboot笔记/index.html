容器功能

组件添加

@Configuration及其对应两种加载模式

现在有如下实例：

public class User {
    String name;
    Integer age;
//getter、setter等方法
}

要将其作为组件添加入程序中，如果是在spring中，我们需要在xml文件中对bean进行注册，现在不需要了

我们直接在config包中创建Myconfig类（类名无所谓）对bean进行注册

@Configuration(proxyBeanMethods = true) //告诉SpringBoot这是一个配置类 == 配置文件 && (proxyBeanMethods = true)是否对bean进行代理
public class MyConfig {
    //@Bean给容器中添加组件。以方法名作为组件的id。返回类型就是组件类型。返回的值，就是组件在容器中的实例
    @Bean
    public User user01(){
        return new User("hengxing",22);
    }
}

然后在启动类中写

@SpringBootApplication
public class MainApplication {
    public static void main(String[] args) {
        ConfigurableApplicationContext run = SpringApplication.run(MainApplication.class, args);
        System.out.println("===========================通过容器获取===========================");
        Pet tomcat01 = run.getBean("tomcat01",Pet.class);
        Pet tomcat02 = run.getBean("tomcat01",Pet.class);
        System.out.println("这两个bean相等吗？" + (tomcat01 == tomcat02));

        MyConfig bean = run.getBean(MyConfig.class);
        System.out.println("现在使用的对象是："+bean);

        System.out.println("---------------------调用两次注册方法获取--------------------------------------");
        User user01 = bean.user01();
        User user02 = bean.user01();
        System.out.println("这两个bean相等吗？" + (user01 == user02));
        
    }
}

这里是控制台输出的结果：

===========================分割线===========================
这两个bean相等吗？true
现在使用的对象是：com.hengxing.boot.config.MyConfig$$EnhancerBySpringCGLIB$$f289fa12@5488b5c5
---------------------调用两次注册方法--------------------------------------
这两个bean相等吗？true

对于结果中的信息有三点要解释：

  1. 单实例属性

    在启动类MainApplication第一部分我们获取了两次tomcat01，并进行比对。他们是完全一致的，说明

    配置类里面使用@Bean标注在方法上给容器注册组件，默认是单实例的。

    另外我们在第二部分打印出了bean的组件名，说明

    配置类本身也是组件

  2. 配置类注解中的proxyBeanMethods

    关注bean的组件名，其中EnhancerBySpringCGLIB表明是被springboot增强过的对象。==如何增强呢？==每次你调用构造方法时，springboot会检查容器中是否有当前对象，若有就不再新建，直接使用已有的对象，即==保持组件单实例属性==。

    来看第三部分，我们通过直接调用bean对象的构造方法，对比他们发现在

    (proxyBeanMethods = true) 的情况下，也就是默认情况下，答案是true。而(proxyBeanMethods = false) 的情况下，不进行代理，答案是false。也就是两次取出的对象都是新生成的，而不是容器中的单实例对象。

    而对于第一部分，我们是通过getbean方法从容器中获取的，所以不论是否使用代理，结果都是相同的，因为容器中对象的单实例属性不会改变。

    这里是当(proxyBeanMethods = false) 的情况下控制台的输出

    ===========================通过容器获取===========================
这两个bean相等吗？true
现在使用的对象是：com.hengxing.boot.config.MyConfig@30b9eadd
---------------------调用两次注册方法获取--------------------------------------
这两个bean相等吗？false

    另外你看，这里的bean组件名就不再是增强过的对象了

    这里的true和false分别对应两个模式：

    Full(proxyBeanMethods = true)、【保证每个@Bean方法被调用多少次返回的组件都是单实例的】

    Lite(proxyBeanMethods = false)【每个@Bean方法被调用多少次返回的组件都是新创建的】

什么场景下使用这两个模式呢？
  • 配置 类组件之间无依赖关系用Lite模式加速容器启动过程，减少判断
    • 配置类组件之间有依赖关系，方法会被调用得到之前单实例组件，用Full模式

    在实例类中添加Pet属性

    Integer age;
    Pet pet;
//getter、setter等属性，注意构造方法不要直接把pet属性写进去，要和之前一样

        this.age = age;
    }

    修改MyConfig类中

    @Bean
   public User user01(){
       User hengxing = new User("hengxing", 22);
       //当开启代理时，User组件依赖了Pet组件
       hengxing.setPet(tomcat());

   }
   
@Bean("tomcat01")
   public Pet tomcat(){
       return new Pet("tomcat");
   }

    然后启动类中编写方法

      User hengxing = run.getBean("user01", User.class);
      Pet pet = run.getBean("tomcat01", Pet.class);
      System.out.println("获取的对象这个bean中的Pet，是否为容器中的Pet:" + (hengxing.getPet()==pet));

    在这里，如果开启代理，那么答案就是true

你可以这样理解这个过程，@Configuration这个注释将你的配置类生效并告诉springboot这是个配置类，这个类现在就变成了一个容器。再使用@Bean向这个类中添加组件，定义一个一个Bean方法的过程，就是添加组件的过程。

@Bean、@Component、@Controller、@Service、@Repository

这些和spring中的用法相同

@ComponentScan、@Import

@ComponentScan用来确定组件的扫描路径，也就是去哪里找组件的定义类。

@Import用来导入外部类

这个注解可以给容器中自动创建出所声明的类型的组件、默认组件的名字就是全类名

例如，在配置类中添加

@Import({User.class, DBHelper.class})
@Configuration
public class MyConfig

在启动类中

System.out.println("====================Import检查========================");
String[] names = run.getBeanNamesForType(User.class);//获取所有User类组件的名字
for (String name : names) {
    System.out.println(name);
}
DBHelper bean1 = run.getBean(DBHelper.class);
System.out.println(bean1);

控制台输出为

====================Import检查========================
com.hengxing.boot.bean.User
user01
ch.qos.logback.core.db.DBHelper@5300f14a

第一行输出的是导入的User组件，默认是全类名

第二行user01是@bean方法注入的组件

第三行也是导入的DBHelper组件

@Conditional条件装配

条件装配：满足Conditional指定的条件，则进行组件注入

我们以@ConditionalOnBean为例，这个注解的作用是当容器中有某个组件时加载当前组件。例如：

在配置类中，将user01类设置为容器中有tomcat01这个组件注册时才会注册

public class MyConfig {
    @ConditionalOnBean(name = "tomcat01")  //这一句
    @Bean
    public User user01(){
        User hengxing = new User("hengxing", 22);
        hengxing.setPet(tomcat());
        return hengxing;
    }

    @Bean("tomcat01")
    public Pet tomcat(){
        return new Pet("tomcat");
    }
}

然后在启动类中接着写

System.out.println("===================检查容器中是否有组件===================");
System.out.println("容器中有tomcat01吗？"+run.containsBean("tomcat01"));
System.out.println("容器中有user01吗？"+run.containsBean("user01"));

按照我们当前的理解，加了条件装配后，由于tomcat01存在，所以结果应该是两个true。我们来看一下控制台

===========================检查容器中是否有组件===========================
容器中有tomcat01吗？true
容器中有user01吗？false

怎么不对？这就引出了条件装配的另一个特性：时序性或称顺序性（官方并没有对这个有定义，我自己便于理解起的名字）。其实很好理解，根据代码顺序执行的特性，在向容器中注册user01这个组件时tomcat01这个组件还没有被注册，所以就不会被注册进容器中，所以user01不存在。而之后tomcat01根据@bean注释被注册进容器中，所以tomcat01存在。

如果我们将tomcat01放在user01前进行注册，结果就会是两个true了。

现在，请尝试不断调换两个组件的注册顺序和修改条件装配的语句，加深理解。

配置绑定

如何使用springboot读取到properties文件中的内容，并且把它封装Bean中，以供随时使用；

配置文件中我们要有相关记录：

#properties.yml
car:
  brand: BYD
  color: red

有两种方式：

1.@Component + @ConfigurationProperties

生成bean类文件

@Component
@ConfigurationProperties(prefix = "car")
public class Car {
    String brand;
    String color;
//getter、setter等方法
}

@ConfigurationProperties(prefix = "car")是告诉springboot配置文件中前缀为car的设置，对应文件中的各种属性。Component是声明为一个组件，也就是注入容器中，只有在容器中的组件，才会拥有SpringBoot提供的强大功能。

再写一个controller来拦截访问

@RestController
public class CarController {

    @Autowired
    Car car;

    @RequestMapping("/car")
    public Car recar(){
        return car;
    }

@Autowired可以将容器中的组件引入controller里，下面的就是简单的再页面返回信息而已。这样，你就可以在浏览器中看到返回的结果{"brand":"BYD","color":"red"}

2.@EnableConfigurationProperties + @ConfigurationProperties

类文件去掉@Component

@ConfigurationProperties(prefix = "car")
public class Car {
    String brand;
    String color;
//getter、setter等方法
}

在配置类中加入这句注释@EnableConfigurationProperties(Car.class)

@Configuration(proxyBeanMethods = false)
@EnableConfigurationProperties(Car.class)
public class MyConfig {
    //内容和之前一样
}

这个注解的作用是：

  1. 开启Car配置绑定功能，这样@ConfigurationProperties就可以发挥作用了
  2. 把这个Car这个组件自动注册到容器中，也就是添加到了配置类所注册的容器中

自动配置原理

自动配置类

@SpringBootApplication我们知道他是自动为springboot注册启动类的，通过查看源码我们可以了解到，他是一个复合注解，其结构是这样的

其实相当于三个注解合并，

@SpringBootConfiguration
@EnableAutoConfiguration
@ComponentScan(excludeFilters = { @Filter(type = FilterType.CUSTOM, classes = TypeExcludeFilter.class),
		@Filter(type = FilterType.CUSTOM, classes = AutoConfigurationExcludeFilter.class) })
public @interface SpringBootApplication{}

其中第一个是@SpringBootConfiguration，里面是@Configuration。代表当前是一个配置类。

第三个是@ComponentScan，指定扫描哪里，Spring注解；

第二个是重点，@EnableAutoConfiguration

@AutoConfigurationPackage
@Import(AutoConfigurationImportSelector.class)
public @interface EnableAutoConfiguration {}

  1. @AutoConfigurationPackage

    @Import(AutoConfigurationPackages.Registrar.class)  //给容器中导入一个组件
public @interface AutoConfigurationPackage {}
//利用Registrar给容器中导入一系列组件
//将指定的一个包下的所有组件导入进来？MainApplication 所在包下。

  2. @Import(AutoConfigurationImportSelector.class)

    1、利用getAutoConfigurationEntry(annotationMetadata);给容器中批量导入一些组件
    2、调用List configurations = getCandidateConfigurations(annotationMetadata, attributes)获取到所有需要导入到容器中的配置类
    3、利用工厂加载 Map<String, List> loadSpringFactories(@Nullable ClassLoader classLoader)；得到所有的组件
    4、从META-INF/spring.factories位置来加载一个文件。
    默认扫描我们当前系统里面所有META-INF/spring.factories位置的文件
    例如：spring-boot-autoconfigure-2.3.4.RELEASE.jar包里面也有META-INF/spring.factories

文件里面写死了spring-boot一启动就要给容器中加载的所有配置类
spring-boot-autoconfigure-2.3.4.RELEASE.jar/META-INF/spring.factories

你可以打开这个包查看

# Auto Configure
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
org.springframework.boot.autoconfigure.admin.SpringApplicationAdminJmxAutoConfiguration,\
------------
    省略很多行，一共是从22行到148行，刚好一共127个包
---------------
org.springframework.boot.autoconfigure.webservices.client.WebServiceTemplateAutoConfiguration
按需开启自动配置项

但是springboot真的加载了这么多包吗？并没有。我们在启动类中测试一下到底容器中注册了多少bean，来看控制台结果

===========一共有多少bean被注册？================
132

如果127个包全部被注入容器中，再加上我们刚刚配置的那么多bean肯定会更多的，所以很明显这127个包中有没有被加载的。springboot如何确定这些包不被注入容器呢？

答案是：==利用条件装配机制==

现在我们随便打开一个springboot的组件包，能够在注释中找到类似这样的一句：

@ConditionalOnClass({JobLauncher.class, DataSource.class})，也就是说，如果我们没有注册相应的类进入容器，springboot也不会加载对应的组件。很棒的想法！！！

修改默认配置
@Bean
@ConditionalOnBean(MultipartResolver.class)  //容器中有这个类型组件
@ConditionalOnMissingBean(name = DispatcherServlet.MULTIPART_RESOLVER_BEAN_NAME) //容器中没有这个名字 multipartResolver 的组件
public MultipartResolver multipartResolver(MultipartResolver resolver) {
    //给@Bean标注的方法传入了对象参数，这个参数的值就会从容器中找。
    //SpringMVC multipartResolver。防止有些用户配置的文件上传解析器不符合规范
    // Detect if the user has created a MultipartResolver but named it incorrectly
    return resolver;
}
给容器中加入了文件上传解析器；
总结：

  • SpringBoot先加载所有的自动配置类 xxxxxAutoConfiguration

  • 每个自动配置类按照条件进行生效，默认都会绑定配置文件指定的值。xxxxProperties里面拿。xxxProperties和配置文件进行了绑定

  • 生效的配置类就会给容器中装配很多组件

  • 只要容器中有这些组件，相当于这些功能就有了

  • 定制化配置

    • 用户直接自己@Bean替换底层的组件
    • 用户去看这个组件是获取的配置文件什么值就去修改。

xxxxxAutoConfiguration —> 组件 —> xxxxProperties里面拿值 —-> application.properties

Web开发

静态资源访问

只要静态资源放在类路径下： called /static (or /public or /resources or /META-INF/resources)

访问 ： 当前项目根路径/ + 静态资源名。就可以访问到静态资源

原理： 静态映射/**

请求进来，先去找Controller看能不能处理。不能处理的所有请求又都交给静态资源处理器。静态资源也找不到则响应404页面

改变静态资源访问前缀

spring:
  mvc:
    static-path-pattern: /res/**

这样，以后所有访问静态资源的链接必须前缀/res/静态资源名，例如：http://localhost:8080/res/2.jpg

改变静态资源存储路径

spring:
  web:
    resources:
      static-locations: [classpath:/haha/]

这样静态资源的路径就要在根目录下的haha文件夹下

请求处理-Rest映射

  • Rest风格支持（使用HTTP请求方式动词来表示对资源的操作）

    • 以前：/getUser 获取用户 /deleteUser 删除用户 /editUser 修改用户 /saveUser 保存用户
    • 现在： /user *GET-*获取用户 *DELETE-*删除用户 *PUT-*修改用户 *POST-*保存用户
    • 核心Filter；HiddenHttpMethodFilter
      • 由于一些历史原因，html表单只能传get、post请求，而这个方法会把我们表单传来的post请求进行解析，如果其中有隐藏信息表示它是put、或delete请求就重新封装这个请求
      • 用法： 表单method=post，隐藏域 _method=put
      • SpringBoot中手动开启
    • 扩展：如何把_method 这个名字换成我们自己喜欢的。

我们先来看一下请求是如何过滤的。

@Bean

@ConditionalOnProperty(
        prefix = "spring.mvc.hiddenmethod.filter",
        name = {"enabled"}
    )
    public OrderedHiddenHttpMethodFilter hiddenHttpMethodFilter() {
        return new OrderedHiddenHttpMethodFilter();
    }

//OrderedHiddenHttpMethodFilter()又继承了这个下面方法
public class HiddenHttpMethodFilter extends OncePerRequestFilter {
    private static final List<String> ALLOWED_METHODS;
    public static final String DEFAULT_METHOD_PARAM = "_method";
    private String methodParam = "_method";
    
    //这里还提供了设置参数名的方法
    public void setMethodParam(String methodParam) {
        Assert.hasText(methodParam, "'methodParam' must not be empty");
        this.methodParam = methodParam;
    }

hiddenHttpMethodFilter() 这是隐藏http方法过滤器，在我们没有HiddenHttpMethodFilter.class这个类，并且配置文件中没有配置或enabled值为false时会帮我们生成过滤器。其次，我们看生成方法中有methodParam参数，默认为"_method"。所以，我们只需要修改这个参数就行了，上面的DEFAULT_METHOD_PARAM是final值不可修改。

实现方法很简单，自己注册一个配置类，调用setMethodParam方法修改即可。

@Configuration(proxyBeanMethods = false)
public class WebConfig {

    @Bean
    public HiddenHttpMethodFilter hiddenHttpMethodFilter(){
        HiddenHttpMethodFilter hiddenHttpMethodFilter = new HiddenHttpMethodFilter();
        hiddenHttpMethodFilter.setMethodParam("_m");//这里修改
        return hiddenHttpMethodFilter;
    }
}

进入getHandler方法，大概是遍历所有HandlerMapping

@Nullable
    protected HandlerExecutionChain getHandler(HttpServletRequest request) throws Exception {
        if (this.handlerMappings != null) {
            Iterator var2 = this.handlerMappings.iterator();

            while(var2.hasNext()) {
                HandlerMapping mapping = (HandlerMapping)var2.next();
                HandlerExecutionChain handler = mapping.getHandler(request);
                if (handler != null) {
                    return handler;
                }
            }
        }

打上断点分析，可以看到HandlerMapping：保存了所有@RequestMapping和handler的映射规则，打开第一个查看，内部的mappingRegistry保存了五个mapping信息

总结一下具体过程：

DispatcherServlet拦截到请求之后，会调用doService开始响应，内部使用doDispatch做分发，决定谁来处理信息。该方法中getHandler保存了所有映射规则，根据我们定义好的mapping来返回handler。

矩阵变量

  • 什么是矩阵变量

    /cars/{path;low=34;brand=byd,audi,yd}这样用;隔开的变量就是矩阵变量

  • 有什么用途

    比如在页面开发中，cookie被禁用了，session中的内容要怎么取用？

    例如给session存一个值session.set(a,b)会有一个jsessionid，统一放在cookie中，每次发送请求时携带。现在cookie被禁用我们就没办法读取session的信息了。

    这个时候就可以用矩阵变量解决，这种行为被称为路径重写：/abc;jsessionid=xxx

会看到报错

Resolved [org.springframework.web.bind.MissingMatrixVariableException: Required matrix variable ‘low’ for method parameter type Integer is not present]

这是因为系统会屏蔽掉分号后的内容，所以找不到变量，我们需要把屏蔽分号这个步骤关掉

关键点在这里，webMvc自动配置方法中通过实现自动配置类的接口方法configurePathMatch

public static class WebMvcAutoConfigurationAdapter implements WebMvcConfigurer, ServletContextAware {
    
    @Override
    public void configurePathMatch(PathMatchConfigurer configurer) {
        //省略不重要部分
        if (servletUrlMapping.equals("/") && singleDispatcherServlet()) {
            UrlPathHelper urlPathHelper = new UrlPathHelper();
            urlPathHelper.setAlwaysUseFullPath(true);
            configurer.setUrlPathHelper(urlPathHelper);
        }
    });

其中urlPathHelper类中有setRemoveSemicolonContent方法会移除分号

/**
	 * Set if ";" (semicolon) content should be stripped from the request URI.
	 * <p>Default is "true".
	 */
	public void setRemoveSemicolonContent(boolean removeSemicolonContent) {
		checkReadOnly();
		this.removeSemicolonContent = removeSemicolonContent;
	}

  • 方法一，重写一个路径匹配器，记得我们第一种自定义自动配置的方法吗
